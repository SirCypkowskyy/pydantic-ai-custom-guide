So far you wrote tools yourself, as Python functions living in the same process as the
agent. MCP (Model Context Protocol) flips that around. It is an open standard that lets an
agent talk to an external tool server, no matter what language that server is written in or
where it runs. Your agent becomes an MCP client, and the server hands it a ready-made set
of tools and resources.

This means you do not have to rewrite the integration with a database, a filesystem, or a
third-party API every time. You connect a proven MCP server and its tools show up in your
agent as if you had written them by hand.

## Why MCP at all

Imagine someone already wrote a server that can read files, query a database, or search
the web. MCP defines a shared language for the agent to ask such a server for the list of
available tools and then call them. The client and the server can live in different
languages and on different machines.

<Callout kind="key">
In Pydantic AI you connect an MCP server as a <Term definition="A toolset is a set of tools the agent can use. An MCP server is one such set, and you pass it through the toolsets parameter.">toolset</Term>, that is, through the
`toolsets=[...]` parameter on the agent. From the model's point of view, tools from an MCP
server look exactly like the ones defined with the `@agent.tool` decorator. The model does
not know, and does not need to know, where they come from.
</Callout>

## The two transports to learn first

Pydantic AI ships several MCP client classes. Two of them cover most cases, and those are
the ones to start with.

| Class | Transport | When to use |
| --- | --- | --- |
| `MCPServerStdio` | stdio (subprocess) | a server you launch locally as a command |
| `MCPServerStreamableHTTP` | Streamable HTTP | a remote server reachable at a URL |

`MCPServerStdio` starts the server itself as a subprocess and talks to it over standard
input and output. `MCPServerStreamableHTTP` connects to a server that already runs
somewhere and listens at an HTTP address.[^sse]

## A local server over stdio

The simplest setup: the MCP server is a command that Pydantic AI will launch for you. You
give it the program and its arguments, and the client handles the rest.

```python
import asyncio

from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio

server = MCPServerStdio("python", args=["mcp_server.py"], timeout=10)
agent = Agent("openai:gpt-5.2", toolsets=[server])


async def main() -> None:
    async with agent:
        result = await agent.run("What is the weather in Paris?")
        print(result.output)
        #> The weather in Paris is sunny and 26 degrees Celsius.


asyncio.run(main())
```

Notice the `async with agent`. This block opens the connection to the server, starts the
subprocess, and keeps it alive for the duration of the runs. When you leave the block, the
client cleans up after itself and shuts the subprocess down.

<Callout kind="warning">
Without the `async with` context the MCP server is never started, and the `agent.run` call
will find no tools. If you do several runs in a row, keep them inside a single `async with`
block so you do not start the server over and over.
</Callout>

## A remote server over Streamable HTTP

When the MCP server already runs somewhere as an HTTP service, you pass only its address.
The rest of the code stays the same.

```python
import asyncio

from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStreamableHTTP

server = MCPServerStreamableHTTP("http://localhost:8000/mcp")
agent = Agent("openai:gpt-5.2", toolsets=[server])


async def main() -> None:
    async with agent:
        result = await agent.run("What is 7 plus 5?")
        print(result.output)
        #> The answer is 12.


asyncio.run(main())
```

## How a run with an MCP server flows

The first thing the client does after entering `async with` is ask the server for its list
of tools. Those tools go to the model alongside your own. When the model decides to call a
server tool, Pydantic AI forwards the call over the MCP protocol and returns the result
back into the agent loop.

## Several servers at once

The `toolsets` parameter takes a list, so you can connect several MCP servers at the same
time, plus your own tools next to them. A single `async with agent` covers the lifecycle of
all of them.

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio, MCPServerStreamableHTTP

files = MCPServerStdio("python", args=["files_server.py"], tool_prefix="fs")
weather = MCPServerStreamableHTTP("http://localhost:8000/mcp", tool_prefix="weather")

agent = Agent("openai:gpt-5.2", toolsets=[files, weather])
```

The `tool_prefix` parameter adds a prefix to the tool names from a given server. It is
handy when two servers have a tool with the same name and you want to avoid a collision.

## Building your own MCP server around an agent

So far you have been the client. Now we flip the roles: your code is the MCP server, and
inside the tool sits a full Pydantic AI agent. This is how you expose a capability you built
yourself as an ordinary MCP tool that any client can use.

The simplest way is `FastMCP` from the official MCP SDK. You create a server, decorate a
function with `@server.tool()`, and inside its body you call the agent and return its output.

```python
from mcp.server.fastmcp import FastMCP

from pydantic_ai import Agent

server = FastMCP("Pydantic AI Server")
server_agent = Agent(
    "anthropic:claude-haiku-4-5", instructions="always reply in rhyme"
)


@server.tool()
async def poet(theme: str) -> str:
    """Poem generator"""
    r = await server_agent.run(f"write a poem about {theme}")
    return r.output


if __name__ == "__main__":
    server.run()
```

From the client's point of view this is just a `poet` tool. The client does not know that an
agent, a model, and instructions sit underneath. What matters to it is the tool name, its
arguments, and the returned text.

<Callout kind="key">
Server and client are two roles of the same protocol. In the earlier sections the agent was a
client and reached for someone else's tools. Here the agent is the heart of the server, and
its own capability becomes a tool for others. You can run this same file as a stdio server and
connect to it with the client from the first section of the chapter.
</Callout>

## Sampling, where the server borrows the client's model

In the example above the server had its own model and its own keys. Often you do not want
that: a public server would rather have the client pay for model calls, and a local server
would rather not hold any credentials. That is what sampling is for. Instead of connecting to
a model directly, the server asks the client over the MCP protocol to run a model call and
receives the finished response.

On the server side you use `MCPSamplingModel` as the agent's model and hand it the session
from the tool context.

```python
from mcp.server.fastmcp import Context, FastMCP

from pydantic_ai import Agent
from pydantic_ai.models.mcp_sampling import MCPSamplingModel

server = FastMCP("Pydantic AI Server with sampling")
server_agent = Agent(instructions="always reply in rhyme")


@server.tool()
async def poet(ctx: Context, theme: str) -> str:
    """Poem generator"""
    r = await server_agent.run(
        f"write a poem about {theme}",
        model=MCPSamplingModel(session=ctx.session),
    )
    return r.output


if __name__ == "__main__":
    server.run()
```

On the client side sampling is supported automatically when the client is a Pydantic AI
agent. All the server needs is a `sampling_model`. The easiest way to set it is the
`agent.set_mcp_sampling_model()` call, which sets the agent's model as the sampling model on
every connected MCP server.

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio

server = MCPServerStdio("python", args=["generate_svg.py"])
agent = Agent("openai:gpt-5.2", toolsets=[server])


async def main() -> None:
    async with agent:
        agent.set_mcp_sampling_model()
        result = await agent.run("Create an image of a robot in a punk style.")
        print(result.output)
```

You can also turn sampling off by setting `allow_sampling=False` when you create the server
reference. The client will then refuse model calls to the server even if it asks for them.

## The FastMCP client

`FastMCP` is a higher-level MCP framework that adds things beyond the specification itself,
like tool transformation and OAuth. Pydantic AI ships a separate toolset, `FastMCPToolset`,
that connects to servers through the FastMCP client. It is an alternative to the `MCPServer*`
classes from the earlier sections.

If your FastMCP server lives in the same codebase as the agent, you can build the toolset
straight from the server object and save a network round trip.

```python
from fastmcp import FastMCP

from pydantic_ai import Agent
from pydantic_ai.toolsets.fastmcp import FastMCPToolset

fastmcp_server = FastMCP("my_server")


@fastmcp_server.tool()
async def add(a: int, b: int) -> int:
    return a + b


toolset = FastMCPToolset(fastmcp_server)
agent = Agent("openai:gpt-5.2", toolsets=[toolset])


async def main() -> None:
    result = await agent.run("What is 7 plus 5?")
    print(result.output)
    #> The answer is 12.
```

`FastMCPToolset` also accepts a plain server address, for example `FastMCPToolset("http://localhost:8000/mcp")`,
a path to a script, or a JSON configuration.

<Callout kind="warning">
`FastMCPToolset` does not yet support sampling or elicitation. If you need those mechanisms,
stay with the standard `MCPServerStdio`, `MCPServerStreamableHTTP`, or `MCPServerSSE` classes.
</Callout>

## Many servers from a config file, and elicitation

Instead of creating every server by hand in code, you can describe them in a JSON file and
load them with `load_mcp_servers()`. It is handy when you have several servers, or when you
want to steer them without touching code.

```json
{
  "mcpServers": {
    "python-runner": {
      "command": "uv",
      "args": ["run", "mcp-run-python", "stdio"]
    },
    "weather": {
      "command": "python",
      "args": ["mcp_server.py"]
    },
    "weather-api": {
      "url": "http://localhost:3001/sse"
    },
    "calculator": {
      "url": "http://localhost:8000/mcp"
    }
  }
}
```

Entries with a `command` field are stdio servers. Entries with a `url` field are HTTP
servers, where an address ending in `/sse` is treated as the older `MCPServerSSE` transport
and anything else as Streamable HTTP. The configuration also supports environment variable
expansion through `${VAR}` and `${VAR:-default}`.

```python
from pydantic_ai import Agent
from pydantic_ai.mcp import load_mcp_servers

servers = load_mcp_servers("mcp_config.json")
agent = Agent("openai:gpt-5.2", toolsets=servers)


async def main() -> None:
    async with agent:
        result = await agent.run("What is 7 plus 5?")
        print(result.output)
```

Finally, elicitation. This is the mechanism where the server asks the client for missing data
while a tool is running, instead of demanding everything upfront. In practice the model says
"hold on, I need to know X first". On the client side you attach an `elicitation_callback`, a
function that receives the server's request, asks the user, and sends the result back.

```python
from typing import Any

from mcp.client.session import ClientSession
from mcp.shared.context import RequestContext
from mcp.types import ElicitRequestParams, ElicitResult

from pydantic_ai import Agent
from pydantic_ai.mcp import MCPServerStdio


async def handle_elicitation(
    context: RequestContext[ClientSession, Any, Any],
    params: ElicitRequestParams,
) -> ElicitResult:
    print(f"\n{params.message}")
    response = input("Response: ")
    return ElicitResult(action="accept", content={"response": response})


server = MCPServerStdio(
    "python", args=["restaurant_server.py"], elicitation_callback=handle_elicitation
)
agent = Agent("openai:gpt-5.2", toolsets=[server])
```

<Callout kind="note">
Elicitation works only for flat objects with string, number, boolean, or enum fields. The
callback can return `action="accept"` with the data, `action="decline"` when the user says
no, or `action="cancel"` when the user aborts. Never ask the user for sensitive data this way.
</Callout>
