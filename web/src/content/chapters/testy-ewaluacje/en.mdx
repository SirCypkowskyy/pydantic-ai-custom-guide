Agents need tests like any other code, but a real model costs money, runs slowly, and never
returns the same answer twice. That is why Pydantic AI ships two stand-in models that run
locally, with no network and no API key. For unit tests you swap the real model for a stub,
and for measuring quality you reach for a separate package, `pydantic-evals`.

This chapter separates two things that are easy to confuse. Tests check that the code around
the agent is correct: that tools get called, that the output has the right shape. Evals
measure whether the model's answers are any good. The first should be fast and
deterministic; the second is soft by nature.

## Why we do not test against the real model

A unit test should be fast, repeatable, and free. A real model breaks all three rules at
once. <Term definition="TestModel is a built-in stub model that does not connect to the network. It produces simple deterministic responses and automatically calls every tool the agent has.">TestModel</Term>
fixes this by pretending to be a model: it never leaves the process, it calls each of the
agent's tools in turn, and it returns a predictable output. That way you test the control
flow, not the model's intelligence.

<Callout kind="key">
Tests and evals answer different questions. A test asks "does my code do what it should". An
eval asks "are the model's answers good enough". The first must be deterministic; the second
cannot be made so, so keep them in separate places.
</Callout>

## TestModel and agent.override

The heart of testing is `agent.override(model=...)`. It swaps the model inside a `with`
block, so production code stays untouched and the test agent uses the stub. You do not need
to change the agent definition or pass the model in as an argument.

```python
from pydantic_ai import Agent
from pydantic_ai.models.test import TestModel

agent = Agent("openai:gpt-5.2", instructions="Answer concisely.")


def test_agent_runs_offline() -> None:
    with agent.override(model=TestModel()):
        result = agent.run_sync("Whatever I type here never hits the network.")
    # TestModel returns a simple, deterministic response.
    assert isinstance(result.output, str)
    assert result.output
```

`TestModel` automatically calls every tool registered on the agent, guessing arguments from
the type hints. That verifies the tools can be called at all and that their signatures hold
together. If you set `output_type` to a Pydantic model, `TestModel` builds a sample object
of that type.

## FunctionModel when you need control

`TestModel` is convenient but dumb: you do not control what it answers. When you want to
force a specific response or a specific tool call, you reach for
<Term definition="FunctionModel lets you write a plain Python function that decides what the model replies based on the messages it receives. It gives full control over behavior in a test.">FunctionModel</Term>.
You write a function that receives the messages and builds the model response itself.

```python
from pydantic_ai import Agent
from pydantic_ai.messages import ModelMessage, ModelResponse, TextPart
from pydantic_ai.models.function import AgentInfo, FunctionModel

agent = Agent("openai:gpt-5.2")


def always_fixed(messages: list[ModelMessage], info: AgentInfo) -> ModelResponse:
    return ModelResponse(parts=[TextPart(content="A fixed answer for the test.")])


def test_forced_response() -> None:
    with agent.override(model=FunctionModel(always_fixed)):
        result = agent.run_sync("Question that does not matter.")
    assert result.output == "A fixed answer for the test."
```

A rule of thumb: use `TestModel` when you only want the agent to walk the whole loop. Use
`FunctionModel` when you have to reproduce a specific scenario, for example a model that
first calls a tool and then summarizes the result.

## capture_run_messages

Sometimes asserting on `result.output` alone is not enough. You want to check that the agent
actually called a given tool, or how many rounds went to the model. `capture_run_messages`
records every message exchanged during a run.

```python
from pydantic_ai import Agent, capture_run_messages
from pydantic_ai.messages import ToolCallPart
from pydantic_ai.models.test import TestModel

agent = Agent("openai:gpt-5.2")


@agent.tool_plain
def square_area(side: int) -> int:
    return side * side


def test_tool_was_called() -> None:
    with capture_run_messages() as messages:
        with agent.override(model=TestModel()):
            agent.run_sync("Compute the area of a square with side 6.")

    calls = [
        part
        for message in messages
        for part in message.parts
        if isinstance(part, ToolCallPart)
    ]
    assert any(call.tool_name == "square_area" for call in calls)
```

<Callout kind="tip">
`capture_run_messages` also helps when debugging errors. When an agent raises
`UnexpectedModelBehavior`, the captured messages show exactly where the tool loop got stuck
and what arguments the model tried to pass.
</Callout>

## Evals with pydantic-evals

Tests say "the code works". They do not say "the answers are good". That is the job of a
separate package, `pydantic-evals`. With it you build a <Term definition="A Dataset is a collection of test cases (Case) together with evaluators. You run it against a function or agent and get back a report of scores.">Dataset</Term>
made of cases (`Case`) and evaluators, then run it against your function.

```python
from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import Evaluator, EvaluatorContext, IsInstance


class MatchedCity(Evaluator[str, str]):
    def evaluate(self, ctx: EvaluatorContext[str, str]) -> float:
        if ctx.output == ctx.expected_output:
            return 1.0
        if ctx.expected_output.lower() in ctx.output.lower():
            return 0.8
        return 0.0


dataset = Dataset(
    name="capitals",
    cases=[
        Case(
            name="france",
            inputs="What is the capital of France?",
            expected_output="Paris",
            metadata={"difficulty": "easy"},
        ),
    ],
    evaluators=[IsInstance(type_name="str"), MatchedCity()],
)


async def guess_city(question: str) -> str:
    return "Paris"


report = dataset.evaluate_sync(guess_city)
report.print(include_input=True, include_output=True)
```

`evaluate_sync` runs each case through the given function and passes the result through the
evaluators. Each evaluator returns a numeric score or an assertion, and `report.print`
assembles it into a table. The scored function can be a plain function or a thin wrapper that
calls `agent.run` and returns `result.output`.[^golden]

## LLMJudge for soft criteria

Sometimes you cannot write an exact assertion because a "good answer" is a matter of tone or
completeness. Then you use `LLMJudge`, which scores the output against a rubric you write in
plain English. This is the one moment in testing where we deliberately go back to a real
model, because the model is the judge.

```python
from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import LLMJudge

dataset = Dataset(
    name="customer_support",
    cases=[
        Case(
            name="refund_request",
            inputs={"query": "I want my money back", "order_id": "12345"},
            evaluators=[
                LLMJudge(
                    rubric=(
                        "Response should acknowledge the request empathetically, "
                        "ask for the reason, and mention the 30-day refund policy."
                    ),
                    include_input=True,
                ),
            ],
        ),
    ],
)
```

<Callout kind="warning">
`LLMJudge` calls a real model, so it costs money and can be flaky. Keep it in a slower eval
suite, not in the fast unit tests you run on every commit. Mixing the two worlds slows the
whole test grid down.
</Callout>

## Where each thing belongs

You run the fast, deterministic tests on every commit. You run evals, especially the ones
with `LLMJudge`, less often, because they are slow and they cost money. That split keeps the
day-to-day work cheap while still keeping the model's quality under control.

## The wider shelf of built-in evaluators

`IsInstance` and `LLMJudge` are only the start. `pydantic-evals` ships a whole set of ready
made case-level evaluators that run for free and instantly, with no network. `EqualsExpected`
checks exact equality with `expected_output`. `Equals(value=...)` compares against a specific
value. `Contains(value=...)` checks for a substring, a list element, or a key-value pair in a
dict. `MaxDuration(seconds=...)` makes sure a case stays inside a time budget. You combine
them so the cheap deterministic checks run first and the expensive `LLMJudge` runs last.

```python
from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import (
    Contains,
    IsInstance,
    LLMJudge,
    MaxDuration,
)

dataset = Dataset(
    name="combined_evaluators",
    cases=[Case(name="refund", inputs="I want my money back")],
    evaluators=[
        # Fast, deterministic checks first.
        IsInstance(type_name="str"),
        Contains(value="30 days", case_sensitive=False),
        MaxDuration(seconds=2.0),
        # The expensive LLM judge only at the end.
        LLMJudge(rubric="Response is helpful and mentions the refund policy."),
    ],
)
```

<Callout kind="tip">
Order matters in practice. Cheap checks like `IsInstance` or `Contains` catch a broken shape
or a missing keyword right away, so you do not waste money on `LLMJudge` for cases that are
obviously wrong anyway.
</Callout>

## Report evaluators for the whole experiment

Regular evaluators run once per case. Report evaluators work differently: they run once after
every case has been evaluated, and they receive the full
<Term definition="An EvaluationReport is the complete result of an experiment: every case with its inputs, outputs, expected outputs, numeric scores, and assertions. A report evaluator computes statistics over the whole set.">EvaluationReport</Term>.
That lets you compute experiment-wide statistics: a confusion matrix, a precision-recall
curve with area under the curve, or plain accuracy. You pass them through the
`report_evaluators` parameter on `Dataset`.

```python
from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import ConfusionMatrixEvaluator


def classifier(text: str) -> str:
    text = text.lower()
    if "cat" in text or "meow" in text:
        return "cat"
    if "dog" in text or "bark" in text:
        return "dog"
    return "unknown"


dataset = Dataset(
    name="animal_classifier",
    cases=[
        Case(name="cat", inputs="The cat goes meow", expected_output="cat"),
        Case(name="dog", inputs="The dog barks", expected_output="dog"),
    ],
    report_evaluators=[
        ConfusionMatrixEvaluator(
            predicted_from="output",
            expected_from="expected_output",
            title="Animal Classification",
        ),
    ],
)

report = dataset.evaluate_sync(classifier)
# report.analyses contains a ConfusionMatrix result.
```

The built-in report evaluators are `ConfusionMatrixEvaluator` (confusion matrix),
`PrecisionRecallEvaluator` (precision-recall curve with AUC), `ROCAUCEvaluator`, and
`KolmogorovSmirnovEvaluator`. You can also write your own by inheriting from
`ReportEvaluator` and returning a `ScalarResult` or a `TableResult`. When Logfire is
configured, the analyses attach to the experiment span and render in the UI as heatmaps and
charts.

## Span-based evaluators over OTel spans

Sometimes a correct output is not enough. The agent may have reached the right answer by
accident, without calling the tool it was supposed to call, or after five rounds too many.
Span-based evaluation checks **how** the agent reached the result, not just **what** it
returned. It requires `logfire` to be installed and configured, because it works on the
OpenTelemetry spans captured during the run.

```python
import logfire

from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import HasMatchingSpan

logfire.configure(send_to_logfire="if-token-present")

dataset = Dataset(
    name="behavior_check",
    cases=[Case(inputs="Find order 12345")],
    evaluators=[
        # The search tool must be called.
        HasMatchingSpan(
            query={"name_contains": "search_tool"},
            evaluation_name="used_search",
        ),
        # The database query must stay under 100 ms.
        HasMatchingSpan(
            query={"and_": [
                {"name_contains": "database"},
                {"max_duration": 0.1},
            ]},
            evaluation_name="fast_database",
        ),
        # The dangerous tool must not be called.
        HasMatchingSpan(
            query={"not_": {"name_contains": "delete_database"}},
            evaluation_name="safe_execution",
        ),
    ],
)
```

`HasMatchingSpan` takes a `SpanQuery`, a dictionary of conditions. You match by name
(`name_contains`, `name_equals`, `name_matches_regex`), by attributes (`has_attributes`), by
timing (`min_duration`, `max_duration`), and even by parent-child relationships
(`some_child_has`, `min_descendant_count`). You combine conditions with `and_`, `or_`, and
`not_`. For more complex analysis you write your own evaluator and reach for `ctx.span_tree`,
which lets you walk the whole span tree and compute, for example, the total time spent in
model calls.

Span-based evaluation tests a behavioral contract, not just an input-output relationship. It
catches regressions where the output is still correct but the internal logic broke, for
example an agent that stopped using search and now answers from memory.
</Callout>

## Dataset management and serialization

A dataset does not have to live only in code. `pydantic-evals` serializes a `Dataset` to YAML
or JSON and loads it back, so your golden dataset sits in the repository like any other file.
Saving also writes a JSON schema file, which gives you field autocomplete and live validation
in the editor. The full round-trip preserves cases, evaluators, and report evaluators too.

```python
from typing import Any

from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import EqualsExpected, IsInstance

dataset = Dataset[str, str, Any](
    name="golden_suite",
    cases=[
        Case(name="france", inputs="Capital of France?", expected_output="Paris"),
    ],
    evaluators=[IsInstance(type_name="str"), EqualsExpected()],
)

# Save to YAML, golden_suite_schema.json is written alongside.
dataset.to_file("golden_suite.yaml")

# Load it back, the format is inferred from the extension.
loaded = Dataset[str, str, Any].from_file("golden_suite.yaml")

# Add cases on the fly.
loaded.add_case(name="poland", inputs="Capital of Poland?", expected_output="Warsaw")
```

Built-in evaluators are recognized by name automatically. If you use a custom evaluator, you
pass its type through `custom_evaluator_types=[...]` when loading, so `pydantic-evals` knows
how to reconstruct it. You can also generate a whole dataset with a model through
`generate_dataset`, providing the input and output schemas as Pydantic models.

<Callout kind="tip">
Keep separate datasets for separate purposes: a fast smoke test of critical paths, a thorough
suite for detailed scoring, and a narrow regression suite for specific past bugs. Each one is
a plain YAML file that is easy to review in a pull request.
</Callout>

## Multi-run evaluation and online scoring

A real model never returns the same answer twice, so a single pass over a dataset says little
about stability. By running the same dataset many times you see the spread of scores and tell
apart the cases the model solves confidently from the ones it gets right only sometimes. The
same idea extends to production: because span-based evaluation operates on the same telemetry
data you already collect in production, you can run the same assertions over live traffic,
scoring the model online and not just against a frozen dataset.

```python
import statistics

from pydantic_evals import Case, Dataset
from pydantic_evals.evaluators import LLMJudge

dataset = Dataset(
    name="stability_scoring",
    cases=[Case(name="refund", inputs="I want my money back")],
    evaluators=[LLMJudge(rubric="Response is empathetic and mentions the refund policy.")],
)


async def respond(query: str) -> str:
    return "I understand. Under our policy you have 30 days to return it."


# A handful of runs reveals the variance one pass cannot show.
scores: list[float] = []
for _ in range(5):
    report = dataset.evaluate_sync(respond)
    passed = sum(
        1
        for case in report.cases
        for assertion in case.assertions.values()
        if assertion.value
    )
    scores.append(passed / max(len(report.cases), 1))

print(f"mean={statistics.mean(scores):.2f} stdev={statistics.pstdev(scores):.2f}")
```

High variance is a signal that the answer is unstable and worth tightening the instructions
or adding edge cases. In production you do not have an `expected_output` for every request,
so you lean on evaluators that need no reference: `LLMJudge` with a rubric, phrase-presence
checks, or span assertions. That closes the loop: the same criteria that guard quality on
every commit guard it on live traffic too.
