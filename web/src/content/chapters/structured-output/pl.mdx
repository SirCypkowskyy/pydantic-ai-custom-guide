Domyślnie agent zwraca zwykły tekst. To wygodne na demo, ale w prawdziwej aplikacji
chcesz danych, które da się zapisać do bazy, przekazać do funkcji albo pokazać w UI bez
zgadywania, co model miał na myśli. W tym rozdziale zmusimy model do zwracania struktury
zgodnej z modelem Pydantic, a Pydantic AI zadba o walidację i ewentualne poprawki.

Cała sztuczka sprowadza się do jednego parametru: `output_type`. Podajesz model Pydantic,
a `result.output` przestaje być stringiem i staje się gotowym, otypowanym obiektem.

## Od tekstu do struktury

Wystarczy zdefiniować <Term definition="Model Pydantic to klasa dziedzicząca po BaseModel, która opisuje pola i ich typy. Pydantic waliduje dane wejściowe względem tej definicji.">model Pydantic</Term> i przekazać go jako `output_type`.
Model dostaje schemat JSON wygenerowany z tej klasy i odpowiada danymi, które do niego pasują.

```python
from pydantic import BaseModel

from pydantic_ai import Agent


class Lokalizacja(BaseModel):
    miasto: str
    kraj: str


agent = Agent("openai:gpt-5.2", output_type=Lokalizacja)

result = agent.run_sync("Gdzie odbyły się igrzyska olimpijskie w 2012 roku?")
print(result.output)
#> miasto='Londyn' kraj='Wielka Brytania'
print(type(result.output))
#> <class '__main__.Lokalizacja'>
```

`result.output` jest teraz instancją `Lokalizacja`, a nie tekstem. Twój edytor i type
checker wiedzą, że ma pola `miasto` i `kraj`, oba typu `str`. To ta sama generyczność,
o której była mowa przy agencie: `Agent[..., Lokalizacja]` niesie typ wyniku ze sobą.

<Callout kind="key">
Model Pydantic służy tutaj do dwóch rzeczy naraz. Po pierwsze, jego schemat JSON mówi
modelowi językowemu, jak ma wyglądać odpowiedź. Po drugie, ta sama definicja waliduje to,
co faktycznie wróciło. Jedno źródło prawdy dla promptu i dla walidacji.
</Callout>

## Co się dzieje pod spodem

Kiedy podasz `output_type`, do uruchomienia dochodzi dodatkowy krok. Pydantic AI bierze
odpowiedź modelu, próbuje ją sparsować i zwalidować względem schematu. Jeśli się uda,
dostajesz obiekt. Jeśli nie, agent ma jeszcze szansę poprawić błąd.

## Walidacja i ModelRetry

Walidacja Pydantic działa tu tak samo jak wszędzie indziej. Jeśli model zwróci pole w złym
typie albo pominie wymagane pole, walidacja się nie powiedzie. Pydantic AI nie poddaje się
od razu: odsyła komunikat o błędzie z powrotem do modelu i prosi o poprawioną odpowiedź.
Liczbę takich prób kontrolujesz parametrem `retries`.

Czasem chcesz dołożyć własną regułę, której nie da się wyrazić samymi typami. Do tego służy
`@agent.output_validator`. Jeśli dane są nie do przyjęcia, podnieś `ModelRetry` z opisem
problemu, a model dostanie ten tekst i spróbuje jeszcze raz.

```python
from pydantic import BaseModel

from pydantic_ai import Agent, ModelRetry


class Rezerwacja(BaseModel):
    liczba_osob: int
    sala: str


agent = Agent(
    "openai:gpt-5.2",
    output_type=Rezerwacja,
    retries=2,
)


@agent.output_validator
def sala_musi_istniec(rez: Rezerwacja) -> Rezerwacja:
    dostepne = {"A", "B", "C"}
    if rez.sala not in dostepne:
        raise ModelRetry(
            f"Sala {rez.sala!r} nie istnieje. Wybierz jedną z: {sorted(dostepne)}."
        )
    return rez


result = agent.run_sync("Zarezerwuj salę B dla 4 osób.")
print(result.output)
#> liczba_osob=4 sala='B'
```

<Callout kind="warning">
`ModelRetry` ma sens tylko przy błędach, które model może naprawić, na przykład zła nazwa
sali albo wartość spoza dozwolonego zakresu. Przy awarii bazy danych czy braku uprawnień
podnoś zwykły wyjątek. Ponawianie nic tam nie da, a tylko spali tokeny.
</Callout>

## Trzy tryby wyjścia

Pod spodem Pydantic AI może wymusić strukturę na trzy sposoby. Domyślnie wybiera najlepszy
dla danego modelu, ale możesz to nadpisać, owijając typ jednym z markerów.

| Tryb | Marker | Jak działa | Kiedy |
| --- | --- | --- | --- |
| Tool | `ToolOutput` | model „wywołuje narzędzie" o schemacie wyniku | domyślny, szerokie wsparcie |
| Native | `NativeOutput` | natywny structured output dostawcy | gdy model to wspiera, najbardziej niezawodny |
| Prompted | `PromptedOutput` | schemat wstrzyknięty w prompt jako instrukcja | modele bez wsparcia narzędzi i structured output |

```python
from pydantic import BaseModel

from pydantic_ai import Agent, NativeOutput


class Lokalizacja(BaseModel):
    miasto: str
    kraj: str


# Wymuszamy natywny structured output zamiast domyślnego trybu narzędziowego.
agent = Agent(
    "openai:gpt-5.2",
    output_type=NativeOutput(Lokalizacja),
)

result = agent.run_sync("Gdzie leży wieża Eiffla?")
print(result.output)
#> miasto='Paryż' kraj='Francja'
```

Marker zmienia tylko mechanizm, nie sam typ wyniku.[^modes] `result.output` dalej jest
instancją `Lokalizacja`. Zacznij od wartości domyślnej i sięgaj po `NativeOutput` albo
`PromptedOutput` tylko wtedy, gdy konkretny model tego wymaga.

## Unie i wiele typów

`output_type` przyjmuje też listę typów albo unię. To wygodne, gdy agent może zwrócić jedną
z kilku różnych odpowiedzi. Model sam wybiera pasujący wariant, a ty dostajesz otypowany
obiekt, który możesz rozróżnić przez `isinstance`.

```python
from pydantic import BaseModel

from pydantic_ai import Agent


class Sukces(BaseModel):
    wartosc: float


class Blad(BaseModel):
    powod: str


agent = Agent("openai:gpt-5.2", output_type=[Sukces, Blad])

result = agent.run_sync("Ile to 3 podzielone przez 0?")
print(result.output)
#> powod='Nie można dzielić przez zero.'
```

<Callout kind="tip">
Para „sukces albo błąd" to wygodny wzorzec. Zamiast łapać wyjątek albo parsować tekst,
agent zwraca jeden z dwóch jasno opisanych kształtów, a resztą zajmuje się normalny kod
Pythona po stronie wywołania.
</Callout>

## Funkcje wyjścia

Zamiast modelu Pydantic możesz przekazać do `output_type` zwykłą funkcję. Pydantic AI
zwaliduje argumenty wygenerowane przez model, wywoła twoją funkcję i jej wynik stanie się
wyjściem uruchomienia. To wygodne, gdy chcesz dodatkowo przetworzyć dane, sięgnąć po coś
z bazy albo przekazać zadanie innemu agentowi. Funkcja wyjścia działa podobnie do narzędzia,
z trzema różnicami: model jest zmuszony jedną z nich wywołać, wywołanie kończy uruchomienie,
a wynik nie wraca do modelu.

Funkcja może opcjonalnie przyjąć `RunContext` jako pierwszy argument i może podnieść
`ModelRetry`, żeby poprosić model o poprawione argumenty.

```python
import re

from pydantic import BaseModel

from pydantic_ai import Agent, ModelRetry


class Wiersz(BaseModel):
    nazwa: str
    kraj: str


tabele = {
    "stolice": [
        Wiersz(nazwa="Amsterdam", kraj="Holandia"),
        Wiersz(nazwa="Meksyk", kraj="Meksyk"),
    ]
}


def wykonaj_zapytanie(zapytanie: str) -> list[Wiersz]:
    """Wykonaj zapytanie SQL na bazie danych."""

    dopasowanie = re.match(r"SELECT \* FROM (\w+)", zapytanie)
    if not dopasowanie:
        raise ModelRetry(f"Nieobsługiwane zapytanie: {zapytanie!r}.")

    tabela = dopasowanie.group(1)
    if tabela not in tabele:
        raise ModelRetry(
            f"Nieznana tabela {tabela!r}. Dostępne: {', '.join(tabele)}."
        )
    return tabele[tabela]


agent = Agent(
    "openai:gpt-5.2",
    output_type=wykonaj_zapytanie,
    instructions="Jesteś agentem SQL. Zamieniaj zapytania na SQL i wykonuj je.",
)

result = agent.run_sync("Podaj wszystkie stolice.")
print(result.output)
#> [Wiersz(nazwa='Amsterdam', kraj='Holandia'), Wiersz(nazwa='Meksyk', kraj='Meksyk')]
```

Domyślnie funkcja przyjmująca stringa też dostaje narzędzie wyjścia. Jeśli wolisz, żeby model
podał tekst zwykłą odpowiedzią, a twoja funkcja go przetworzyła, owiń ją markerem `TextOutput`.

```python
from pydantic_ai import Agent, TextOutput


def na_slowa(tekst: str) -> list[str]:
    return tekst.split()


agent = Agent("openai:gpt-5.2", output_type=TextOutput(na_slowa))

result = agent.run_sync("Kim był Albert Einstein?")
print(result.output)
#> ['Albert', 'Einstein', 'był', 'niemieckim', 'fizykiem', 'teoretycznym.']
```

<Callout kind="tip">
Funkcja wyjścia to naturalny sposób na delegowanie zadań między agentami. Funkcja routera
może wewnątrz wywołać `await inny_agent.run(...)` i zwrócić jego wynik, a model w ogóle nie
musi wiedzieć, że pod spodem działa drugi agent.
</Callout>

## Własny schemat JSON i kontekst walidacji

Czasem nie da się opisać wyjścia modelem Pydantic, bo schemat JSON przychodzi z zewnątrz albo
powstaje dynamicznie w trakcie działania programu. Wtedy użyj `StructuredDict()`. Dostajesz
podklasę `dict[str, Any]` z dołączonym schematem, który Pydantic AI przekaże modelowi.

```python
from pydantic_ai import Agent, StructuredDict

OsobaDict = StructuredDict(
    {
        "type": "object",
        "properties": {
            "imie": {"type": "string"},
            "wiek": {"type": "integer"},
        },
        "required": ["imie", "wiek"],
    },
    name="Osoba",
    description="Osoba z imieniem i wiekiem.",
)

agent = Agent("openai:gpt-5.2", output_type=OsobaDict)
result = agent.run_sync("Stwórz osobę.")
print(result.output)
#> {'imie': 'Jan Kowalski', 'wiek': 30}
```

<Callout kind="warning">
Przy `StructuredDict()` Pydantic AI nie waliduje odebranego obiektu. Wynik to zwykły słownik
i to twój kod musi go ostrożnie odczytać na wypadek, gdyby model się pomylił. Walidację możesz
dorzucić w `@agent.output_validator` i podnieść `ModelRetry`, żeby model spróbował jeszcze raz.
</Callout>

Kiedy walidacja potrzebuje dodatkowych danych spoza samej odpowiedzi, przekaż je przez parametr
`validation_context`. Trafi on do walidatorów Pydantic zarówno przy wyjściu, jak i przy
argumentach narzędzi. Co ważne, ten kontekst nie ląduje w promptach wysyłanych do modelu, służy
tylko walidacji po stronie Pythona.

```python
from pydantic import BaseModel, ValidationInfo, field_validator

from pydantic_ai import Agent


class Wartosc(BaseModel):
    x: int

    @field_validator("x")
    def dodaj_przesuniecie(cls, value: int, info: ValidationInfo):
        return value + (info.context or 0)


agent = Agent(
    "openai:gpt-5.2",
    output_type=Wartosc,
    validation_context=10,
)
result = agent.run_sync("Podaj wartość 5.")
print(repr(result.output))  # 5 od modelu + 10 z kontekstu walidacji
#> Wartosc(x=15)
```

## Wyjście opcjonalne i obrazy

Niektóre agenty wykonują całą pracę przez narzędzia i nie muszą zwracać żadnego wyniku
końcowego. Część modeli zwraca wtedy pustą odpowiedź, co domyślnie wymusza ponowienie. Jeśli
chcesz, żeby pusta odpowiedź była uznana za udane uruchomienie, dołącz `None` do `output_type`.

```python
from pydantic_ai import Agent

agent = Agent("anthropic:claude-opus-4-6", output_type=str | None)


@agent.tool_plain
def oznacz_zadanie(zadanie_id: int) -> str:
    """Oznacz zadanie jako wykonane."""
    return f"Zadanie {zadanie_id} oznaczone jako wykonane."


result = agent.run_sync("Oznacz zadanie 1 jako wykonane i nic nie mów.")
print(result.output)
#> None
```

`output_type=str | None` to przypadek wzorcowy: traktowany jak zwykły tekst, gdzie jedynym
sygnałem `None` jest pusta odpowiedź. Sam `None` bez innego typu nie jest poprawny, zawsze
musisz podać przynajmniej jeden dodatkowy typ.

Część modeli potrafi też generować obrazy. Aby obraz stał się wyjściem agenta, ustaw
`output_type` na `BinaryImage`. Jeśli żadne narzędzie do generowania obrazów nie jest jawnie
włączone, Pydantic AI samo dołączy `ImageGenerationTool`.

```python
from pydantic_ai import Agent, BinaryImage

agent = Agent("openai-responses:gpt-5.2", output_type=BinaryImage)

result = agent.run_sync("Wygeneruj obraz aksolotla.")
assert isinstance(result.output, BinaryImage)
```

<Callout kind="key">
Gdy agent nie zawsze ma generować obraz, użyj unii `BinaryImage | str`. Jeśli model zwróci
oba, pierwszeństwo ma obraz, a tekst znajdziesz w `result.response.text`.
</Callout>

## Przerywanie strumienia

Czasem trzeba zatrzymać strumień przed jego zakończeniem: użytkownik klika „przerwij" w czacie
albo masz już dość danych, żeby podjąć decyzję. Wywołaj `cancel()` na `StreamedRunResult`,
a Pydantic AI poprosi dostawcę o przerwanie generowania i zamknie połączenie.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")


async def main():
    async with agent.run_stream("Napisz długi esej o Pythonie") as result:
        tekst = ""
        async for fragment in result.stream_text(delta=True):
            tekst += fragment
            if len(tekst) > 100:
                await result.cancel()
                break
        print(result.cancelled)
        #> True
        print(result.response.state == "interrupted")
        #> True
```

Po przerwaniu finalny `ModelResponse` dostaje stan `state='interrupted'`, więc dalszy kod
pozna niekompletną odpowiedź. Strumień zdarzeń z `run_stream_events()` nie ma metody `cancel()`,
ale używasz go jako menedżera kontekstu i samo wyjście z bloku `async with` posprząta zasoby.

i zdecyduj sam, czy ją zachować, czy odrzucić. Zużycie tokenów po przerwaniu też jest
częściowe i zależne od dostawcy, więc nie polegaj na nim przy rozliczaniu kosztów.
</Callout>
