Every agent run produces a list of messages. That list is the real record of the conversation:
what you sent to the model, what the model replied, and which tools it called along the way. If
you want the agent to remember earlier turns, you do not invent your own format. You take the
list from one run and hand it to the next.

In this chapter we break the message structure into its parts, build a multi-turn conversation,
and serialise the history to JSON so it can survive a process restart or land in a database.

## What a run returns

The result of `run_sync` (or `await run`) is more than just `output`. It also holds the full
message record. There are two entry points to it, and they are easy to confuse:

- `result.all_messages()` returns the entire history, including messages passed in as input.
- `result.new_messages()` returns only the messages produced by this particular run.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2", instructions="Be a helpful assistant.")

result = agent.run_sync("Tell me a joke.")
print(result.output)
#> A programmer walks into a bar. And again. And again. It was a loop.

for message in result.all_messages():
    print(type(message).__name__)
#> ModelRequest
#> ModelResponse
```

A single run with no tools usually gives you two messages: one `ModelRequest` (what went to the
model) and one `ModelResponse` (what came back).

## Two message types

The whole history is a list of <Term definition="ModelMessage is the shared type for messages in the history. Concretely it is either a ModelRequest or a ModelResponse.">`ModelMessage`</Term> objects, which in practice means
`ModelRequest` or `ModelResponse`. Each message is made of parts, because a single turn can
carry several things at once.

| Message type | Direction | Typical parts |
| --- | --- | --- |
| `ModelRequest` | to the model | `UserPromptPart`, `SystemPromptPart`, `ToolReturnPart` |
| `ModelResponse` | from the model | `TextPart`, `ToolCallPart` |

<Callout kind="note">
Instructions are not a separate message. They attach to a `ModelRequest` as its `instructions`
field and get swapped on every run. That is why passing old history forward does not drag stale
instructions along. An old `system_prompt` stays in the history as a `SystemPromptPart`, which
is the important difference.
</Callout>

The message and part types live in `pydantic_ai.messages`. You reach for them when you want to
build a history by hand or filter it before saving.

## Running a multi-turn conversation

An agent is stateless on its own. Conversation memory is just a list of messages that you pass
through the `message_history` parameter. The model then sees the earlier turns and can refer
back to them.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2", instructions="Be a helpful assistant.")

result1 = agent.run_sync("Tell me a joke.")
print(result1.output)
#> Did you hear about the toothpaste scandal? They called it Colgate.

result2 = agent.run_sync(
    "Explain?",
    message_history=result1.new_messages(),
)
print(result2.output)
#> It is a pun: "Colgate" sounds like "cold gate", the chilly gateway to the scandal.
```

When you pass `message_history`, the agent appends the new question to the end of that list and
sends the whole thing to the model. Without the parameter, the second question "Explain?" would
hang in a vacuum, because the model would have no idea what it refers to.

## Serialising history to JSON

In a real application the conversation does not live in one process's memory. You need to put it
somewhere: a database, a file, a user session. Pydantic AI gives you `ModelMessagesTypeAdapter`
for this, which knows the exact shape of messages and can validate them back.

```python
from pydantic_core import to_jsonable_python

from pydantic_ai import Agent, ModelMessagesTypeAdapter

agent = Agent("openai:gpt-5.2", instructions="Be a helpful assistant.")

result1 = agent.run_sync("Tell me a joke.")

# Turn into plain Python objects, ready for json.dumps or a database write.
history = result1.all_messages()
to_store = to_jsonable_python(history)

# ... later, in another process or after a restart ...
restored = ModelMessagesTypeAdapter.validate_python(to_store)

result2 = agent.run_sync("A different joke, please.", message_history=restored)
print(result2.output)
#> Why do programmers confuse Halloween and Christmas? Because OCT 31 == DEC 25.
```

The `to_jsonable_python` helper from `pydantic_core` breaks messages down into dicts, lists, and
strings that pass cleanly through `json.dumps`. Going the other way,
`ModelMessagesTypeAdapter.validate_python` rebuilds the full `ModelRequest` and `ModelResponse`
objects together with their parts.[^adapter]

<Callout kind="warning">
History grows with every turn, and so does the number of tokens sent to the model. For long
conversations, pass a chosen slice or a summary instead of the whole thing. You can also use the
`history_processors` parameter to trim the list before it is sent.
</Callout>

## Correlating runs with conversation_id

Every `ModelRequest` and `ModelResponse` carries two identifiers. `run_id` is unique per agent
run. `conversation_id` is shared across all runs that build on the same history. A fresh
`conversation_id` is generated on the first run, stamped onto every message that run produces,
and inherited by later runs that you feed those messages back into through `message_history`.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")

result1 = agent.run_sync("Tell me a joke.")
result2 = agent.run_sync("Explain?", message_history=result1.all_messages())

# Both runs belong to the same conversation.
assert result1.conversation_id == result2.conversation_id

# Forking a thread: ignore the existing conversation_id and start a new one.
forked = agent.run_sync(
    "Tell me a different joke.",
    message_history=result1.all_messages(),
    conversation_id="new",
)
assert forked.conversation_id != result1.conversation_id
```

You can also pass your own `conversation_id="<your-id>"`, for example a chat thread ID from your
database. The value `conversation_id="new"` starts a fresh conversation and ignores whatever is
already sitting in `message_history`. That is handy when you want to branch off an existing
thread without forcing the calling code to generate its own identifier.

<Callout kind="note">
On the observability side, `run_id` lands on the OpenTelemetry span as `gen_ai.agent.call.id`,
and `conversation_id` as `gen_ai.conversation.id`. In Logfire or any OpenTelemetry backend you
can therefore stitch together the traces of a whole multi-turn conversation without tracking
anything yourself. As long as the history round-trips, the conversation ID rides along with it.
</Callout>

## Processing the history before it is sent

Sometimes you want to change the history before it reaches the model: filter out sensitive data,
trim the list to save tokens, or simply give the model less context. The `ProcessHistory`
capability does this, and you pass it through `capabilities`. It wraps a function that takes a
list of `ModelMessage` and returns a modified list of the same type. A processor can be
synchronous or asynchronous, and if you supply several, they run in sequence.

```python
from pydantic_ai import Agent, ModelMessage, ModelRequest
from pydantic_ai.capabilities import ProcessHistory


def filter_responses(messages: list[ModelMessage]) -> list[ModelMessage]:
    """Keep only ModelRequest messages, drop every ModelResponse."""
    return [msg for msg in messages if isinstance(msg, ModelRequest)]


def keep_recent(messages: list[ModelMessage]) -> list[ModelMessage]:
    """Trim the history to the last five messages."""
    return messages[-5:] if len(messages) > 5 else messages


agent = Agent(
    "openai:gpt-5.2",
    capabilities=[ProcessHistory(filter_responses), ProcessHistory(keep_recent)],
)
```

A processor can also take `RunContext` as its first parameter when the decision should depend on
the state of the run, for example on the tokens used so far. Then you only filter the history
once it actually grows long.

```python
from pydantic_ai import Agent, ModelMessage, RunContext
from pydantic_ai.capabilities import ProcessHistory


def context_aware_processor(
    ctx: RunContext[None],
    messages: list[ModelMessage],
) -> list[ModelMessage]:
    if ctx.usage.total_tokens > 1000:
        return messages[-3:]  # When usage is high, keep only recent messages.
    return messages


agent = Agent("openai:gpt-5.2", capabilities=[ProcessHistory(context_aware_processor)])
```

The most economical variant is summarising. Instead of throwing old turns away, you hand them to
a cheaper agent that folds them into a short summary and pastes it back at the head of the
history.

```python
from pydantic_ai import Agent, ModelMessage
from pydantic_ai.capabilities import ProcessHistory

# A cheaper model for summarising old messages.
summarize_agent = Agent(
    "openai:gpt-5-mini",
    instructions="Summarize this conversation, omitting small talk. Focus on the technical decisions.",
)


async def summarize_old_messages(messages: list[ModelMessage]) -> list[ModelMessage]:
    if len(messages) > 10:
        oldest = messages[:10]
        summary = await summarize_agent.run(message_history=oldest)
        # The summary plus the last message, which stays untouched.
        return summary.new_messages() + messages[-1:]
    return messages


agent = Agent("openai:gpt-5.2", capabilities=[ProcessHistory(summarize_old_messages)])
```

<Callout kind="warning">
When you slice or summarise the history, keep tool calls and their returns paired. If a
`ToolCallPart` is left without a matching `ToolReturnPart`, the model may return an error.
Processors replace the history in the run state, so if you care about the original, make a copy
of it first.
</Callout>

## Injecting messages mid-run

Tools and code driving a run can push extra content into the conversation while it is still
going. From inside a tool, where you have a `RunContext`, you use `ctx.enqueue(...)`. From
external code that drives the run through `agent.iter()`, you use `agent_run.enqueue(...)`. The
`priority` parameter decides when the content is delivered. `'asap'` (the default) adds it to the
next `ModelRequest`, and if the agent would otherwise finish first, it redirects the run into one
more request. `'when_idle'` waits until the agent is done with its current work and only then
hands the content over as a follow-up task.

```python
from pydantic_ai import Agent, RunContext
from pydantic_ai.messages import SystemPromptPart

agent = Agent("anthropic:claude-opus-4-7")


@agent.tool
def trigger_alert(ctx: RunContext[None]) -> str:
    ctx.enqueue("Alert: production is degraded, prioritize triage.")
    return "alert raised"


@agent.tool
def enter_incident_mode(ctx: RunContext[None]) -> str:
    # A SystemPromptPart lets you adjust the agent's standing instructions mid-run.
    ctx.enqueue(SystemPromptPart(content="You are now in incident mode: be terse and action-oriented."))
    return "incident mode enabled"
```

When you drive a run from the outside, for example by forwarding events from a webhook or a job
queue, you add work through `AgentRun.enqueue`. Content with `'when_idle'` is only drained when
the agent would otherwise reach the end, so the run has to be driven through `agent.run` or
explicitly through `AgentRun.next()`.

```python
from pydantic_ai import Agent
from pydantic_graph import End

agent = Agent("anthropic:claude-opus-4-7")


async def main():
    async with agent.iter("Summarize the latest deploy report") as agent_run:
        # An external system pushes a follow-up while the agent is working.
        agent_run.enqueue(
            "A new error was just reported, include it in the summary.",
            priority="when_idle",
        )
        node = agent_run.next_node
        while not isinstance(node, End):
            node = await agent_run.next(node)
```

<Callout kind="warning">
Reaching the end of a bare `async for node in agent_run:` loop with undrained `'when_idle'`
messages raises `UndrainedPendingMessagesError`, since those messages would otherwise be silently
lost. Each end-of-run redirect opens a new model request, so a tool that enqueues on every step
will loop the run forever. Set `UsageLimits` as a safety net.
</Callout>

## History straight to JSON bytes

`all_messages()` and `new_messages()` have variants that return ready JSON bytes:
`all_messages_json()` and `new_messages_json()`. They help when you want to drop the history into
a `bytes` column in a database or send it back to a frontend, without manually routing it through
`to_jsonable_python` and `json.dumps`.

```python
from pydantic_ai import Agent, ModelMessagesTypeAdapter

agent = Agent("openai:gpt-5.2", instructions="Be a helpful assistant.")

result1 = agent.run_sync("Tell me a joke.")

# Raw JSON bytes, ready for a bytes column or a network response.
as_json = result1.all_messages_json()

# Validation works straight on the bytes, with no intermediate json.loads.
restored = ModelMessagesTypeAdapter.validate_json(as_json)

result2 = agent.run_sync("A different joke, please.", message_history=restored)
print(result2.output)
#> Why do programmers prefer dark mode? Because light attracts bugs.
```

The difference from `to_jsonable_python` in the earlier section is simple. That helper gives you
Python objects that you still serialise yourself. The `_json` variants return bytes directly, and
on the way back `ModelMessagesTypeAdapter.validate_json` rebuilds the full message objects from
them.
