The Agent is the central piece of Pydantic AI. It is a single object that ties a language
model, instructions, tools, and an output type into one reusable whole. If you have used
FastAPI before, this will feel familiar: you define the agent once and run it from many
places in your application.

## What an Agent actually is

A good mental model is a class, or a FastAPI app. You create it once, usually as a
module-level constant, and use it for the lifetime of the program. An Agent gathers
several things at once:

- the model it uses (for example `openai:gpt-5.2`),
- <Term definition="Instructions tell the model how to behave. In Pydantic AI this is usually the instructions parameter.">instructions</Term>, the description of how it should behave,
- optional tools the model can call,
- the <Term definition="The output is the structured result the agent returns. It defaults to plain text but can be a Pydantic model.">output</Term> type you expect,
- a dependencies type, if you use one.

<Callout kind="key">
An Agent is generic in two types: the dependencies type and the output type. Because of
that, your editor and type checker know exactly what to expect from `result.output` and
what lands in `ctx.deps`. This typed nature is why the whole thing is called "the Pydantic
way of building agents".
</Callout>

## Your first agent

The simplest possible agent needs only a model. Everything else has sensible defaults.

```python
from pydantic_ai import Agent

agent = Agent(
    "openai:gpt-5.2",
    instructions="Answer concisely, in a single sentence.",
)

result = agent.run_sync("Where does the name Pydantic AI come from?")
print(result.output)
#> It combines the Pydantic library with building AI agents.
```

The first argument is a model identifier in the `provider:name` format. The `instructions`
parameter describes how the agent should behave. The result has an `output` field, which
here is plain text.

## How a run flows

When you call `run_sync`, a few steps happen in order. The agent builds a list of messages,
sends it to the model, and then processes the response. If tools are involved, the loop can
repeat before a final result appears.

## Four ways to run an agent

Pydantic AI gives you four methods to run an agent. They all take the same set of arguments
and differ only in how they hand back the result.

| Method | Returns | When to use |
| --- | --- | --- |
| `run_sync()` | a finished result | scripts, tests, synchronous code |
| `run()` | a coroutine with the result | async code, such as a server |
| `run_stream()` | a stream of chunks | when you want to show text as it arrives |
| `run_stream_events()` | a stream of events | when you need insight into every step |

```python
import asyncio

from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    # The async version
    result = await agent.run("What is the capital of Poland?")
    print(result.output)
    #> The capital of Poland is Warsaw.

    # Streaming text in real time
    async with agent.run_stream("Name three cities in Poland.") as response:
        async for chunk in response.stream_text():
            print(chunk)


asyncio.run(main())
```

## Instructions versus system prompt

Pydantic AI separates two ideas that are easy to mix up. `instructions` is the default and
recommended choice. `system_prompt` exists mainly for backward compatibility and differs in
one important detail.

<Callout kind="note">
Instructions are not added to the message history passed between runs. A system prompt is.
So when you build a multi-turn conversation and pass earlier messages forward, the current
agent's `instructions` still apply, and old system prompts do not sneak back in.
</Callout>

Both forms can be static or dynamic. You define dynamic ones with a decorator, and they have
access to `RunContext`, so they can depend on data passed through `deps`.

```python
from datetime import date

from pydantic_ai import Agent, RunContext

agent = Agent(
    "openai:gpt-5.2",
    deps_type=str,
    instructions="Address the user by their name.",
)


@agent.instructions
def add_the_name(ctx: RunContext[str]) -> str:
    return f"The user's name is {ctx.deps}."


@agent.instructions
def add_the_date() -> str:
    return f"Today's date is {date.today().isoformat()}."


result = agent.run_sync("What day is it?", deps="Frank")
print(result.output)
#> Hi Frank, today is the 2nd of January.
```

## RunContext in a nutshell

`RunContext` is the object passed to dynamic instructions and tools. It gives access to
dependencies through `ctx.deps`, and to information about the current run, such as token
usage or the messages so far. Dependencies get their own chapter, because they are what make
agents genuinely testable.[^deps]

<Callout kind="tip">
Start with the simplest version: a model plus one sentence of instructions. Add tools, a
typed output, and dependencies only when you actually need them. Each of those gets its own
chapter in this guide.
</Callout>

## Looking inside a run with agent.iter()

Under the hood, every agent uses a graph (pydantic-graph) to manage its execution flow.
A plain `run()` simply traverses that graph from start to finish. When you want to see each
step or inject your own logic between stages, use `agent.iter()`. It returns an `AgentRun`
object that you can async-iterate over as a stream of the graph's nodes.

The nodes have concrete types. `UserPromptNode` prepares the user prompt, `ModelRequestNode`
sends the request to the model, `CallToolsNode` processes the response and may call tools,
and `End` marks the finish with a ready result.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    nodes = []
    async with agent.iter("What is the capital of France?") as agent_run:
        async for node in agent_run:
            nodes.append(type(node).__name__)
    print(nodes)
    #> ['UserPromptNode', 'ModelRequestNode', 'CallToolsNode', 'End']
    print(agent_run.result.output)
    #> The capital of France is Paris.
```

You can also drive the iteration manually. The `AgentRun.next(node)` method runs the given
node, appends it to the run history, and returns the next node to run. This lets you inspect
or modify a node before it executes.

```python
from pydantic_ai import Agent
from pydantic_graph import End

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    async with agent.iter("What is the capital of France?") as agent_run:
        node = agent_run.next_node
        while not isinstance(node, End):
            node = await agent_run.next(node)
    print(agent_run.result.output)
    #> The capital of France is Paris.
```

## Limits and usage tracking

Every run consumes tokens, and if the model gets stuck in a loop of tool calls it can consume
far too many. Pydantic AI gives you a `UsageLimits` structure that you pass with the
`usage_limits` argument to `run`, `run_sync`, and `run_stream`. Once a run finishes, the
statistics live in `result.usage`.

```python
from pydantic_ai import Agent, UsageLimitExceeded, UsageLimits

agent = Agent("openai:gpt-5.2")

result = agent.run_sync(
    "What is the capital of Italy? Answer with just the city.",
    usage_limits=UsageLimits(response_tokens_limit=10),
)
print(result.output)
#> Rome
print(result.usage)
#> RunUsage(input_tokens=62, output_tokens=1, requests=1)
```

The most practical one is `request_limit`, the cap on the number of requests to the model.
It protects you from an infinite loop where the model keeps calling the same tool over and
over. When the limit is crossed, the run raises `UsageLimitExceeded`.

```python
from pydantic_ai import Agent, ModelRetry, UsageLimitExceeded, UsageLimits

agent = Agent(
    "openai:gpt-5.2",
    instructions="After every response, call a tool to produce another one.",
)


@agent.tool_plain
def loop() -> int:
    raise ModelRetry("Please try again.")


try:
    agent.run_sync("Start!", usage_limits=UsageLimits(request_limit=3))
except UsageLimitExceeded as e:
    print(e)
    #> The next request would exceed the request_limit of 3
```

<Callout kind="tip">
If you have registered many tools, set `request_limit` to bound the number of model turns,
and `tool_calls_limit` to cap the number of successful tool executions within a single run.
The tool calls limit is checked before the tools execute.
</Callout>

## When the model behaves unexpectedly

Sometimes the model exceeds the retry limit, or its API returns an error. In those cases the
run raises `UnexpectedModelBehavior`. To understand what went wrong, wrap the run in
`capture_run_messages()`. This context manager gives you access to all the messages exchanged
during the run, even when it ended with an exception.

```python
from pydantic_ai import (
    Agent,
    ModelRetry,
    UnexpectedModelBehavior,
    capture_run_messages,
)

agent = Agent("openai:gpt-5.2")


@agent.tool_plain
def calc_volume(size: int) -> int:
    if size == 42:
        return size**3
    raise ModelRetry("Please try again.")


with capture_run_messages() as messages:
    try:
        result = agent.run_sync("Get me the volume of a box with size 6.")
    except UnexpectedModelBehavior as e:
        print("An error occurred:", e)
        #> An error occurred: Tool 'calc_volume' exceeded max retries count of 1
        print("cause:", repr(e.__cause__))
        #> cause: ModelRetry('Please try again.')
        print("message count:", len(messages))
    else:
        print(result.output)
```

<Callout kind="note">
If you call a run more than once inside a single `capture_run_messages` context, `messages`
will contain the messages from the first call only. For diagnosing a single run that is plenty,
but it is worth keeping in mind.
</Callout>

## Agent specs and direct requests

You do not have to build an agent in code. Agent specs let you describe one declaratively in
a YAML or JSON file: the model, instructions, model settings, and capabilities. You load that
file with a single line through `Agent.from_file`, which separates application code from
configuration. This lets non-developers, such as domain experts, change an agent's behavior
without touching Python.

```yaml
# agent.yaml
model: openai:gpt-5.2
instructions: You are a helpful research assistant.
model_settings:
  max_tokens: 8192
```

```python
from pydantic_ai import Agent

agent = Agent.from_file("agent.yaml")
result = agent.run_sync("Summarize the topic in three sentences.")
print(result.output)
```

The `Agent.from_spec` method accepts a dict or an `AgentSpec` instance and lets you add extra
arguments, such as `deps_type`, that supplement or override the spec. Instructions and
capabilities are then merged: the spec ones come first, then the ones from code.

At the other end of the spectrum sit direct requests from the `direct` module. This is a thin
layer over the model itself, with no agent loop, tools, or automatic output parsing. You reach
for it when you need full control over a single request or you are building your own
abstraction on top of models.

```python
from pydantic_ai import ModelRequest
from pydantic_ai.direct import model_request_sync

model_response = model_request_sync(
    "openai:gpt-5.2",
    [ModelRequest.user_text_prompt("What is the capital of France?")],
)
print(model_response.parts[0].content)
#> The capital of France is Paris.
print(model_response.usage)
#> RequestUsage(input_tokens=56, output_tokens=7)
```

<Callout kind="key">
Choose an `Agent` when you want the full feature set: tools, retrying, typed output, and
dependencies. Reach for `direct.model_request` when you only need a raw request to a model
with a single API that is the same across providers. Specs are the way to keep an agent's
configuration outside of code.
</Callout>
