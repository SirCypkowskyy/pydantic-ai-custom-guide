Agent to centralny element Pydantic AI. To pojedynczy obiekt, który spina model
językowy, instrukcje, narzędzia i typ wyniku w jedną, wielokrotnego użytku całość.
Jeśli pracowałeś wcześniej z FastAPI, poczujesz się jak w domu: agenta definiujesz raz,
a potem uruchamiasz go w wielu miejscach aplikacji.

## Czym właściwie jest Agent

Dobrym porównaniem jest klasa albo aplikacja FastAPI. Tworzysz ją raz, zwykle jako stałą
na poziomie modułu, i korzystasz z niej przez całe życie programu. Agent zbiera w sobie
kilka rzeczy naraz:

- model, z którego korzysta (na przykład `openai:gpt-5.2`),
- <Term definition="Instrukcje to wytyczne dla modelu opisujące, jak ma się zachowywać. W Pydantic AI to zwykle parametr instructions.">instrukcje</Term>, czyli opis zachowania,
- opcjonalne narzędzia, które model może wywołać,
- typ <Term definition="Output to ustrukturyzowany wynik, który agent ma zwrócić. Domyślnie jest to zwykły tekst, ale może to być model Pydantic.">wyniku</Term>, którego oczekujesz,
- typ zależności, jeśli z nich korzystasz.

<Callout kind="key">
Agent jest generyczny względem dwóch typów: typu zależności i typu wyniku. Dzięki temu
edytor i type checker wiedzą dokładnie, czego się spodziewać po `result.output` oraz co
trafia do `ctx.deps`. Ta typowana natura to powód, dla którego całość nazywa się
„Pydantic w wersji dla agentów".
</Callout>

## Pierwszy agent

Najprostszy możliwy agent potrzebuje tylko modelu. Reszta ma sensowne wartości domyślne.

```python
from pydantic_ai import Agent

agent = Agent(
    "openai:gpt-5.2",
    instructions="Odpowiadaj zwięźle, jednym zdaniem.",
)

result = agent.run_sync("Skąd wzięła się nazwa Pydantic AI?")
print(result.output)
#> Nazwa łączy bibliotekę Pydantic z budową agentów AI.
```

Pierwszy argument to identyfikator modelu w formacie `dostawca:nazwa`. Parametr
`instructions` opisuje, jak agent ma się zachowywać. Wynik wracający z uruchomienia ma
pole `output`, które tutaj jest zwykłym tekstem.

## Jak przebiega uruchomienie

Kiedy wołasz `run_sync`, dzieje się kilka kroków po kolei. Agent buduje listę wiadomości,
wysyła ją do modelu, a następnie przetwarza odpowiedź. Jeśli w grę wchodzą narzędzia,
pętla może się powtórzyć, zanim pojawi się ostateczny wynik.

## Cztery sposoby uruchamiania

Pydantic AI daje cztery metody uruchomienia agenta. Wszystkie przyjmują ten sam zestaw
argumentów i różnią się tylko sposobem, w jaki oddają wynik.

| Metoda | Zwraca | Kiedy używać |
| --- | --- | --- |
| `run_sync()` | gotowy wynik | skrypty, testy, kod synchroniczny |
| `run()` | korutyna z wynikiem | kod asynchroniczny, na przykład serwer |
| `run_stream()` | strumień fragmentów | gdy chcesz pokazywać tekst na bieżąco |
| `run_stream_events()` | strumień zdarzeń | gdy potrzebujesz wglądu w każdy krok |

```python
import asyncio

from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    # Wersja asynchroniczna
    result = await agent.run("Jaka jest stolica Polski?")
    print(result.output)
    #> Stolicą Polski jest Warszawa.

    # Strumieniowanie tekstu w czasie rzeczywistym
    async with agent.run_stream("Wymień trzy miasta w Polsce.") as response:
        async for fragment in response.stream_text():
            print(fragment)


asyncio.run(main())
```

## Instrukcje kontra system prompt

Pydantic AI rozróżnia dwa pojęcia, które łatwo pomylić. `instructions` to domyślny i
zalecany wybór. `system_prompt` istnieje głównie ze względu na zgodność wsteczną i różni
się jednym ważnym szczegółem.

<Callout kind="note">
Instrukcje nie trafiają do historii wiadomości przekazywanej między uruchomieniami.
System prompt owszem. Dlatego gdy budujesz rozmowę wieloturową i przekazujesz dalej
poprzednie wiadomości, `instructions` z bieżącego agenta nadal obowiązują, a stare
system prompty nie wracają niechcianie.
</Callout>

Obie wersje mogą być statyczne albo dynamiczne. Dynamiczne definiujesz dekoratorem i mają
dostęp do `RunContext`, więc mogą zależeć od danych przekazanych w `deps`.

```python
from datetime import date

from pydantic_ai import Agent, RunContext

agent = Agent(
    "openai:gpt-5.2",
    deps_type=str,
    instructions="Zwracaj się do użytkownika po imieniu.",
)


@agent.instructions
def dodaj_imie(ctx: RunContext[str]) -> str:
    return f"Użytkownik ma na imię {ctx.deps}."


@agent.instructions
def dodaj_date() -> str:
    return f"Dzisiejsza data to {date.today().isoformat()}."


result = agent.run_sync("Jaki dziś dzień?", deps="Franek")
print(result.output)
#> Cześć Franek, dziś jest 2 stycznia.
```

## RunContext w pigułce

`RunContext` to obiekt przekazywany do dynamicznych instrukcji i narzędzi. Daje dostęp do
zależności przez `ctx.deps`, a także do informacji o bieżącym uruchomieniu, takich jak
użycie tokenów czy dotychczasowe wiadomości. Zależnościom poświęcamy osobny rozdział, bo
to one sprawiają, że agenty da się sensownie testować.[^deps]

<Callout kind="tip">
Zacznij od najprostszej wersji: model plus jedno zdanie instrukcji. Narzędzia, typowany
wynik i zależności dokładaj dopiero wtedy, gdy naprawdę ich potrzebujesz. Każdy z tych
elementów ma w tym przewodniku własny rozdział.
</Callout>

## Zaglądanie do środka uruchomienia z agent.iter()

Pod spodem każdy agent korzysta z grafu (pydantic-graph), który steruje przebiegiem
uruchomienia. Zwykłe `run()` po prostu przechodzi ten graf od początku do końca. Kiedy
chcesz zobaczyć każdy krok albo wpiąć własną logikę pomiędzy etapami, użyj `agent.iter()`.
Zwraca on obiekt `AgentRun`, po którym można iterować jako po asynchronicznym iteratorze
węzłów grafu.

Węzły mają konkretne typy. `UserPromptNode` przygotowuje zapytanie użytkownika,
`ModelRequestNode` wysyła żądanie do modelu, `CallToolsNode` przetwarza odpowiedź i ewentualnie
wywołuje narzędzia, a `End` oznacza koniec z gotowym wynikiem.

```python
from pydantic_ai import Agent

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    wezly = []
    async with agent.iter("Jaka jest stolica Francji?") as agent_run:
        async for node in agent_run:
            wezly.append(type(node).__name__)
    print(wezly)
    #> ['UserPromptNode', 'ModelRequestNode', 'CallToolsNode', 'End']
    print(agent_run.result.output)
    #> Stolicą Francji jest Paryż.
```

Możesz też przejmować sterowanie ręcznie. Metoda `AgentRun.next(node)` wykonuje podany
węzeł, dopisuje go do historii uruchomienia i zwraca następny węzeł do wykonania. Dzięki
temu możesz obejrzeć lub zmienić węzeł, zanim się wykona.

```python
from pydantic_ai import Agent
from pydantic_graph import End

agent = Agent("openai:gpt-5.2")


async def main() -> None:
    async with agent.iter("Jaka jest stolica Francji?") as agent_run:
        node = agent_run.next_node
        while not isinstance(node, End):
            node = await agent_run.next(node)
    print(agent_run.result.output)
    #> Stolicą Francji jest Paryż.
```

</Callout>

## Limity i śledzenie użycia

Każde uruchomienie zużywa tokeny, a jeśli model wpadnie w pętlę wywołań narzędzi, może
ich zużyć zdecydowanie za dużo. Pydantic AI daje strukturę `UsageLimits`, którą
przekazujesz argumentem `usage_limits` do metod `run`, `run_sync` i `run_stream`. Po
zakończeniu uruchomienia statystyki znajdziesz w `result.usage`.

```python
from pydantic_ai import Agent, UsageLimitExceeded, UsageLimits

agent = Agent("openai:gpt-5.2")

result = agent.run_sync(
    "Jaka jest stolica Włoch? Odpowiedz samą nazwą miasta.",
    usage_limits=UsageLimits(response_tokens_limit=10),
)
print(result.output)
#> Rzym
print(result.usage)
#> RunUsage(input_tokens=62, output_tokens=1, requests=1)
```

Najbardziej praktyczny jest `request_limit`, czyli limit liczby żądań do modelu. Chroni
on przed nieskończoną pętlą, w której model w kółko wywołuje to samo narzędzie. Gdy limit
zostanie przekroczony, uruchomienie podnosi wyjątek `UsageLimitExceeded`.

```python
from pydantic_ai import Agent, ModelRetry, UsageLimitExceeded, UsageLimits

agent = Agent(
    "openai:gpt-5.2",
    instructions="Po każdej odpowiedzi wywołaj narzędzie, by wygenerować kolejną.",
)


@agent.tool_plain
def petla() -> int:
    raise ModelRetry("Spróbuj jeszcze raz.")


try:
    agent.run_sync("Start!", usage_limits=UsageLimits(request_limit=3))
except UsageLimitExceeded as e:
    print(e)
    #> The next request would exceed the request_limit of 3
```

<Callout kind="tip">
Jeśli zarejestrowałeś dużo narzędzi, ustaw `request_limit`, by ograniczyć liczbę tur
modelu, oraz `tool_calls_limit`, by ograniczyć liczbę udanych wywołań narzędzi w jednym
uruchomieniu. Limit wywołań narzędzi sprawdzany jest przed ich wykonaniem.
</Callout>

## Gdy model zachowa się nieoczekiwanie

Czasem model przekroczy limit ponowień albo jego API zwróci błąd. W takich sytuacjach
uruchomienie podnosi wyjątek `UnexpectedModelBehavior`. Żeby zrozumieć, co poszło nie tak,
owiń uruchomienie w `capture_run_messages()`. Ten menedżer kontekstu daje dostęp do
wszystkich wiadomości wymienionych w trakcie uruchomienia, nawet gdy zakończyło się
wyjątkiem.

```python
from pydantic_ai import (
    Agent,
    ModelRetry,
    UnexpectedModelBehavior,
    capture_run_messages,
)

agent = Agent("openai:gpt-5.2")


@agent.tool_plain
def policz_objetosc(size: int) -> int:
    if size == 42:
        return size**3
    raise ModelRetry("Spróbuj jeszcze raz.")


with capture_run_messages() as messages:
    try:
        result = agent.run_sync("Podaj objętość pudełka o boku 6.")
    except UnexpectedModelBehavior as e:
        print("Wystąpił błąd:", e)
        #> Wystąpił błąd: Tool 'policz_objetosc' exceeded max retries count of 1
        print("przyczyna:", repr(e.__cause__))
        #> przyczyna: ModelRetry('Spróbuj jeszcze raz.')
        print("liczba wiadomości:", len(messages))
    else:
        print(result.output)
```

<Callout kind="note">
Jeśli w jednym kontekście `capture_run_messages` wywołasz uruchomienie więcej niż raz,
`messages` będzie zawierać wiadomości wyłącznie z pierwszego wywołania. Do diagnostyki
jednego uruchomienia to wystarcza, ale warto o tym pamiętać.
</Callout>

## Specyfikacje agentów i żądania bezpośrednie

Agenta nie musisz budować w kodzie. Specyfikacje agentów pozwalają opisać go deklaratywnie
w pliku YAML lub JSON: model, instrukcje, ustawienia modelu i capabilities. Taki plik
wczytujesz jedną linijką przez `Agent.from_file`, a kod aplikacji zostaje oddzielony od
konfiguracji. Dzięki temu osoby spoza zespołu programistów, na przykład eksperci dziedzinowi,
mogą zmieniać zachowanie agenta bez dotykania Pythona.

```yaml
# agent.yaml
model: openai:gpt-5.2
instructions: Jesteś pomocnym asystentem badawczym.
model_settings:
  max_tokens: 8192
```

```python
from pydantic_ai import Agent

agent = Agent.from_file("agent.yaml")
result = agent.run_sync("Streść mi temat w trzech zdaniach.")
print(result.output)
```

Metoda `Agent.from_spec` przyjmuje słownik albo instancję `AgentSpec` i pozwala dorzucić
dodatkowe argumenty, na przykład `deps_type`, które uzupełniają lub nadpisują specyfikację.
Instrukcje i capabilities są wtedy łączone: najpierw te ze specyfikacji, potem te z kodu.

Na drugim końcu spektrum stoją żądania bezpośrednie z modułu `direct`. To cienka warstwa
nad samym modelem, bez pętli agenta, narzędzi czy automatycznego parsowania wyniku. Używasz
ich, gdy potrzebujesz pełnej kontroli nad pojedynczym żądaniem albo budujesz własną
abstrakcję na modelach.

```python
from pydantic_ai import ModelRequest
from pydantic_ai.direct import model_request_sync

model_response = model_request_sync(
    "openai:gpt-5.2",
    [ModelRequest.user_text_prompt("Jaka jest stolica Francji?")],
)
print(model_response.parts[0].content)
#> Stolicą Francji jest Paryż.
print(model_response.usage)
#> RequestUsage(input_tokens=56, output_tokens=7)
```

<Callout kind="key">
Wybierz `Agent`, gdy chcesz pełnej funkcjonalności: narzędzia, ponawianie, typowany wynik
i zależności. Sięgnij po `direct.model_request`, gdy potrzebujesz tylko surowego żądania do
modelu z jednolitym API niezależnym od dostawcy. Specyfikacje to z kolei sposób na trzymanie
samej konfiguracji agenta poza kodem.
</Callout>
